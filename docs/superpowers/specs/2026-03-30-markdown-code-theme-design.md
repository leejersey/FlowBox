# Markdown 代码块主题切换设计

**日期：** 2026-03-30

## 目标

在 `Markdown 转网页` 页面中，为代码块增加亮色与暗色主题的即时切换能力，并确保：

- 预览区代码块样式随当前主题立即变化
- “复制富文本”导出的 HTML 使用当前主题的内联样式
- 粘贴到微信公众号编辑器后仍保持当前主题外观
- 下次重新打开页面时，自动恢复上次选择的主题

## 现状

- 当前代码块样式与 token 颜色硬编码在 [MarkdownPage.tsx](/Users/lizexi/Documents/AI/AiCode/FlowBox/src/pages/MarkdownPage.tsx)。
- 当前仅存在一套暗色主题。
- 代码高亮逻辑、token 颜色映射、代码块容器样式都耦合在同一文件中。
- 页面已经具备“复制富文本”能力，复制链路依赖代码块的内联样式，以适配微信公众号编辑器。

## 设计范围

本次只覆盖 `Markdown 转网页` 页面：

- 页面预览区的 fenced code block
- 页面预览区的 inline code
- “复制富文本”导出的代码块样式
- 页面内主题切换与主题记忆

不在本次范围：

- 设置页中的全局代码主题配置
- Butler、剪贴板、灵感页等其他页面的代码块主题统一
- 任意自定义颜色编辑器
- 第三方重型语法高亮库替换

## 方案选择

评估过三种方式：

### 1. 直接在 `MarkdownPage.tsx` 中写两套主题

优点：

- 改动最少
- 落地最快

缺点：

- 主题定义和渲染继续耦合
- 后续扩展第三套主题或复用到其他页面成本高

### 2. 抽出独立主题对象并由页面切换使用

优点：

- 主题与渲染分层清晰
- 页面内即时切换简单直接
- 复制链路和预览链路可以稳定共用一套主题数据
- 后续新增主题或扩展到其他页面不需要重构主渲染逻辑

缺点：

- 比直接硬编码多一个主题模块

### 3. 改为 CSS 变量主题

优点：

- 本地预览切换自然

缺点：

- 微信公众号编辑器不认本地 CSS 变量
- 复制时仍要把 CSS 变量展开成内联样式
- 实现复杂度更高，但对当前目标收益有限

**结论：** 采用方案 2。

## 架构设计

### 1. 主题抽象

将代码块主题从 `MarkdownPage.tsx` 中拆出到独立模块，例如：

- `/Users/lizexi/Documents/AI/AiCode/FlowBox/src/features/markdown/codeThemes.ts`

该模块负责提供两套固定主题：

- `dark`
- `light`

每个主题对象统一定义：

- 代码块容器背景色、边框色、阴影
- 默认文本颜色
- 行号颜色
- inline code 背景、边框、文字色
- token 颜色映射

这样做的目标是把“如何识别 token”和“token 用什么颜色显示”彻底分离。

### 2. 高亮逻辑与主题分层

保留现有轻量高亮逻辑：

- 语言归一化
- 语言推断
- token 拆分
- 注释识别

这些逻辑继续输出 token 类型，例如：

- `plain`
- `comment`
- `keyword`
- `string`
- `number`
- `operator`
- `builtin`
- `heading`

然后由当前主题负责把 token 类型映射到具体颜色。

这样亮色主题只是“换皮”，不会改变当前暗色高亮行为。

### 3. 页面状态

`MarkdownPage` 内新增一个页面级状态：

- `codeTheme: 'light' | 'dark'`

该状态只属于当前页面，不进入全局设置页，但会记忆到本地存储。

## 交互设计

### 1. 切换入口

在 `Markdown 转网页` 页面头部增加一个可见的主题切换器，建议放在“复制富文本”附近。

控件形式使用简单二段切换：

- `亮色`
- `暗色`

不使用下拉框，避免增加交互层级。

### 2. 即时反馈

用户切换主题后：

- 右侧网页预览中的 fenced code block 立即切换主题
- inline code 同步切换主题
- 后续点击“复制富文本”时，使用当前主题生成内联样式 HTML

原则只有一个：

**预览如何显示，复制出去就如何显示。**

### 3. 主题记忆

页面在切换主题后，将当前值写入本地持久化存储，例如：

- `markdown.codeTheme = light`
- `markdown.codeTheme = dark`

页面下次打开时自动恢复上次选择。

这属于页面级持久化，不视为设置页中的全局默认配置。

## 亮色主题目标

亮色主题按用户给出的参考图实现，视觉目标如下：

- 背景接近白底或极浅暖灰
- 边框使用低对比浅灰
- 普通文本使用深灰而非纯黑
- 行号使用更浅的灰色
- 注释使用浅灰并保留斜体
- 关键词使用橙色系
- 字符串使用偏红色系
- 函数或内建调用使用蓝色系
- 默认标识符维持深灰

第一版不做开放式自定义配色，仅提供一套固定亮色主题。

## 数据流

### 1. 预览链路

Markdown 文本进入 `ReactMarkdown` 渲染时：

1. 代码块识别逻辑判断是 inline code 还是 fenced code block
2. 高亮逻辑输出 token 类型
3. 当前主题为容器样式和 token 提供颜色
4. 最终渲染为带内联样式的预览代码块

### 2. 复制链路

点击“复制富文本”时：

1. 读取当前页面主题状态
2. 使用当前主题生成代码块对应的内联 HTML 样式
3. 将完整 HTML 写入剪贴板

复制动作只读取点击当下的主题值，不依赖异步中的中间状态，避免复制出混合主题样式。

## 错误处理

### 1. 默认回退

如果页面初始化时未找到主题值，默认使用 `dark`。

### 2. 脏值处理

如果本地存储中的主题值不是 `light | dark`，自动回退到 `dark`，不能因此导致页面渲染失败。

### 3. 无语言代码块

没有显式语言标记的代码块继续走当前语言推断逻辑。主题切换不影响语言识别，只影响颜色映射。

### 4. 快速切换后立即复制

复制逻辑必须以用户点击复制时的最终主题状态为准，确保导出的 HTML 只包含单一主题样式。

## 测试策略

### 1. 单元测试

- 亮色主题对象应包含完整的容器、inline code 和 token 配置
- 暗色主题对象应保持现有暗色设计不回退
- 页面初始化时应正确恢复上次主题
- 非法主题值应回退为 `dark`

### 2. 渲染测试

- 相同 Markdown 在 `light` 和 `dark` 主题下应生成不同的代码块样式
- inline code 应随主题同步变化
- fenced code block 应随主题同步变化

### 3. 复制链路测试

- `light` 主题下复制出的 HTML 不应包含暗色背景
- `dark` 主题下复制出的 HTML 不应混入亮色 token 颜色
- 导出的 HTML 应使用当前主题的 token 内联颜色

### 4. 人工验证

- 页面内切换亮色与暗色时预览立即变化
- 关闭并重新打开页面后恢复上次选择
- 复制到微信公众号编辑器后，亮色代码块保持浅色外观和高亮
- 暗色代码块保持现有效果不回退

## 成功标准

以下四项全部满足，视为本次功能完成：

- 页面内可以即时切换亮色与暗色代码主题
- 页面再次打开时自动恢复上次选择
- 预览与复制富文本使用同一主题
- 粘贴到微信公众号编辑器后，代码块仍保持当前主题外观
